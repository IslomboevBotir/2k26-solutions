# Практическое задание: Системные требования и архитектура

## Тема проекта: Платформа ИИ-суммаризации заметок (AI Note Summarizer Platform)

Веб-платформа, которая позволяет пользователям загружать заметки (текст, документы, аудиозаписи или расшифровки лекций) и автоматически генерирует краткие резюме, 
ключевые тезисы и учебные материалы с помощью искусственного интеллекта. Платформа предназначена для студентов, специалистов и исследователей, 
которым необходимо быстро обрабатывать большие объёмы текстовой информации.

---

## Этап 1. Требования

### Функциональные требования

1. **Регистрация и аутентификация пользователей**
   Система должна позволять пользователям создавать учётную запись через email/пароль или через OAuth-провайдеров (Google, GitHub). 
   Авторизованные пользователи могут сохранять, просматривать и управлять своими заметками и резюме через личный кабинет.

2. **Загрузка и ввод заметок**
   Пользователи должны иметь возможность вводить заметки в нескольких форматах: набирать или вставлять текст напрямую, загружать документы (PDF, DOCX, TXT), 
   а также загружать аудиофайлы (MP3, WAV), которые автоматически расшифровываются перед суммаризацией.

3. **Суммаризация на основе ИИ**
   Система должна генерировать резюме загруженных заметок с использованием большой языковой модели (LLM). 
   Пользователь может выбрать стиль резюме (краткий обзор, подробное резюме, ключевые тезисы в виде списка) и желаемую длину (например, 100, 300 или 500 слов).

4. **Генерация учебных материалов**
   Помимо резюме, система должна формировать дополнительные учебные материалы: карточки (вопрос/ответ), тесты с выбором ответа и список ключевых терминов с определениями.

5. **Управление заметками и экспорт**
   Пользователи должны иметь возможность организовывать заметки по папкам или тегам, выполнять поиск по сохранённым резюме и экспортировать результаты в нескольких форматах 
   (PDF, Markdown, DOCX) для использования офлайн или отправки другим.

### Нефункциональные требования

1. **Производительность**
   Система должна генерировать резюме для стандартного текстового документа (до 5 000 слов) менее чем за 15 секунд. 
   Расшифровка аудио должна выполняться со скоростью не менее 1 минуты аудио за 10 секунд реального времени.

2. **Масштабируемость**
   Платформа должна поддерживать минимум 10 000 одновременных пользователей и горизонтально масштабироваться при пиковых нагрузках (например, в период сессии) 
   без снижения качества обслуживания.

3. **Безопасность и конфиденциальность**
   Все пользовательские данные должны шифроваться при передаче (HTTPS/TLS 1.3) и при хранении (AES-256). 
   Заметки пользователей не должны использоваться для обучения моделей ИИ без явного согласия, а система должна соответствовать требованиям GDPR.

4. **Доступность (Availability)**
   Платформа должна поддерживать аптайм не менее 99,5%, иметь механизмы автоматического переключения при сбоях (failover) и ежедневное резервное копирование. 
   Плановые технические работы должны проводиться в непиковые часы.

5. **Удобство использования (Usability)**
   Интерфейс должен быть адаптивным и доступным на ПК, планшетах и мобильных устройствах. Система должна поддерживать как минимум три языка 
   (английский, русский, узбекский) и соответствовать стандартам доступности WCAG 2.1 уровня AA.

## Этап 2. Архитектура

### Вариант 1. Монолитная архитектура

В этом подходе все компоненты приложения (управление пользователями, работа с заметками, оркестрация ИИ-обработки и логика хранения) реализуются в рамках единого развёртываемого приложения.
Внешние ИИ-сервисы вызываются через API, но основная логика приложения объединена.

#### Схема системы

```
┌─────────────────────────────────────────────────────────────┐
│                      КЛИЕНТ (Браузер)                       │
│               React / Vue.js Single Page App                │
└───────────────────────────┬─────────────────────────────────┘
                            │ HTTPS
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                  БАЛАНСИРОВЩИК НАГРУЗКИ (Nginx)             │
└───────────────────────────┬─────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│              МОНОЛИТНОЕ BACKEND-ПРИЛОЖЕНИЕ                  │
│                 (Node.js / Django / Spring)                 │
│                                                             │
│  ┌───────────────┐  ┌───────────────┐  ┌────────────────┐   │
│  │ Модуль Auth   │  │ Модуль Заметок│  │  Модуль ИИ     │   │
│  └───────────────┘  └───────────────┘  └────────────────┘   │
│  ┌───────────────┐  ┌───────────────┐  ┌────────────────┐   │
│  │Модуль Экспорта│  │Модуль Поиска  │  │Модуль Биллинга │   │
│  └───────────────┘  └───────────────┘  └────────────────┘   │
└────────┬────────────────────┬─────────────────────┬─────────┘
         │                    │                     │
         ▼                    ▼                     ▼
┌─────────────────┐  ┌─────────────────┐  ┌──────────────────┐
│  PostgreSQL БД  │  │ Файловое хран.  │  │  Внешний ИИ API  │
│ (пользователи,  │  │  (AWS S3 /      │  │  (OpenAI /       │
│ заметки, резюме)│  │ локальный диск) │  │   Anthropic)     │
└─────────────────┘  └─────────────────┘  └──────────────────┘
```

#### Преимущества

- **Простота разработки и развёртывания:** единая кодовая база облегчает начальную настройку, локальную разработку и настройку CI/CD-пайплайнов.
- **Низкие затраты на инфраструктуру:** на старте достаточно одного сервера приложения, что снижает затраты на хостинг и DevOps.
- **Простота отладки и тестирования:** вся логика работает в одном процессе, поэтому поиск ошибок, использование отладчиков и написание интеграционных тестов проще.
- **Быстрая начальная разработка:** небольшие команды могут быстро выпускать новые функции без необходимости проектировать контракты между сервисами.
- **Транзакционная согласованность:** все операции с данными могут выполняться в одной транзакции одной БД, что упрощает обеспечение целостности данных.

#### Недостатки

- **Ограниченная масштабируемость:** приложение приходится масштабировать целиком, даже если нагрузка приходится только на одну функцию (например, ИИ-суммаризацию).
- **Единая точка отказа:** если монолит падает, вся платформа становится недоступной — включая авторизацию, заметки и экспорт.
- **Сложность поддержки при росте:** с увеличением кодовой базы нескольким командам становится трудно работать над ней без конфликтов.
- **Технологическая привязка:** всё приложение привязано к одному языку и фреймворку, что затрудняет внедрение новых технологий для отдельных компонентов.
- **Медленные циклы развёртывания:** любое небольшое изменение требует повторного развёртывания всего приложения, что повышает риски.

### Вариант 2. Микросервисная архитектура

В этом подходе система разделена на независимые сервисы, каждый из которых отвечает за свою предметную область. Сервисы взаимодействуют через API Gateway, 
а для асинхронных задач (ИИ-обработка и расшифровка аудио) используется очередь сообщений.

#### Схема системы

```
┌─────────────────────────────────────────────────────────────┐
│                  КЛИЕНТ (Браузер / Мобильное)               │
└───────────────────────────┬─────────────────────────────────┘
                            │ HTTPS
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                   API GATEWAY (Kong / Nginx)                │
│         (маршрутизация, auth, rate limiting, кеш)           │
└──┬──────────────┬──────────────┬──────────────┬─────────────┘
   │              │              │              │
   ▼              ▼              ▼              ▼
┌────────┐   ┌─────────┐    ┌─────────┐   ┌──────────┐
│  Auth  │   │ Сервис  │    │ Сервис  │   │ Сервис   │
│Сервис  │   │заметок  │    │экспорта │   │биллинга  │
└───┬────┘   └────┬────┘    └────┬────┘   └─────┬────┘
    │             │              │               │
    ▼             ▼              ▼               ▼
┌────────┐  ┌──────────┐   ┌──────────┐   ┌──────────┐
│  БД    │  │   БД     │   │  Файлы   │   │ БД       │
│юзеров  │  │ заметок  │   │   (S3)   │   │ платежей │
└────────┘  └──────────┘   └──────────┘   └──────────┘
                 │
                 ▼
      ┌───────────────────────┐
      │  ОЧЕРЕДЬ СООБЩЕНИЙ    │
      │  (RabbitMQ / Kafka)   │
      └──────┬────────────┬───┘
             │            │
             ▼            ▼
    ┌────────────────┐  ┌────────────────────┐
    │ Сервис ИИ-     │  │ Сервис расшифровки │
    │ суммаризации   │  │      аудио         │
    │  (вызов LLM)   │  │  (Whisper / API)   │
    └────────┬───────┘  └──────────┬─────────┘
             │                     │
             ▼                     ▼
    ┌───────────────────────────────────────┐
    │      Внешние ИИ-провайдеры            │
    │   (OpenAI, Anthropic, Whisper)        │
    └───────────────────────────────────────┘
```

#### Преимущества

- **Независимое масштабирование:** сервис ИИ-суммаризации (самый ресурсоёмкий) можно масштабировать отдельно при росте нагрузки.
- **Изоляция сбоев:** если сервис экспорта падает, пользователи всё ещё могут создавать и суммаризировать заметки. Сбои локализованы.
- **Технологическая гибкость:** каждый сервис может использовать наиболее подходящую технологию (например, Python для ИИ, Go для API Gateway, Node.js для заметок).
- **Параллельная разработка:** разные команды могут работать над разными сервисами, не мешая друг другу.
- **Асинхронная обработка:** тяжёлые задачи (расшифровка аудио, длинные резюме) выполняются в фоне через очереди, что сохраняет отзывчивость интерфейса.

#### Недостатки

- **Высокая сложность:** требуется экспертиза в распределённых системах, service discovery, оркестрации контейнеров (Kubernetes) и мониторинге.
- **Более высокие затраты на инфраструктуру:** нужны дополнительные серверы, базы данных и инструменты (Kubernetes, очереди сообщений, системы наблюдаемости).
- **Сетевые издержки и задержки:** сервисы взаимодействуют по сети, что добавляет задержки и новые точки отказа.
- **Сложная отладка:** отслеживание одного пользовательского запроса через несколько сервисов требует инструментов распределённой трассировки (Jaeger, Zipkin).
- **Проблемы согласованности данных:** транзакции, охватывающие несколько сервисов, требуют сложных паттернов (например, Saga).

### Какой вариант лучше и почему

**Для платформы ИИ-суммаризации заметок лучшим выбором является микросервисная архитектура (Вариант 2)**, однако с важной оговоркой относительно сроков перехода.

Специфика платформы сильно склоняется в пользу микросервисов по следующим причинам.

Компоненты ИИ-суммаризации и расшифровки аудио **принципиально отличаются** от остальной системы. Они требовательны к CPU и памяти, имеют непредсказуемое время обработки 
(двухчасовая аудиолекция обрабатывается гораздо дольше, чем короткая текстовая заметка) и зависят от внешних API, которые могут работать медленно или иметь ограничения. 
Вынесение их в отдельные сервисы с собственной очередью сообщений предотвращает блокировку лёгких операций — например, входа в систему или просмотра сохранённых заметок.

Кроме того, нефункциональные требования предполагают **10 000 одновременных пользователей** и **аптайм 99,5%**. Монолит потребовал бы масштабирования всего приложения целиком, 
даже если нагрузка приходится только на ИИ-компонент, что и дорого, и неэффективно. Микросервисы позволяют точечное масштабирование, что более экономично на таком уровне нагрузки.

Асинхронный сценарий работы (пользователь загружает аудио → очередь → расшифровка → очередь → суммаризация → уведомление) естественно ложится на событийно-ориентированную микросервисную 
архитектуру и выглядел бы искусственно внутри монолита.

**Однако на практике** я бы рекомендовал начинать с **модульного монолита** (Вариант 1 с чёткими внутренними границами модулей) на этапе MVP и **постепенно выделять микросервисы** 
(начиная с сервисов ИИ-суммаризации и расшифровки аудио) по мере роста пользовательской базы. Такой прагматичный подход — иногда называемый стратегией "monolith-first" — позволяет 
избежать тяжёлой операционной сложности микросервисов до того, как продукт подтвердит соответствие рынку, сохраняя при этом возможность плавного перехода, когда масштаб действительно 
этого потребует.

**Для целевой архитектуры на этапе полноценного продакшена микросервисы — правильный долгосрочный выбор** для данной платформы.
