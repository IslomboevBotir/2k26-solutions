# Требования и архитектура системы

## Тема проекта: Wedding Planning Portal (Портал планирования свадьбы)

---

## Этап 1. Требования

### Функциональные требования

1. **Управление гостями**  
   Система должна позволять пользователям создавать, редактировать и удалять список гостей, а также отслеживать статус приглашений (отправлено, подтверждено, отклонено) и отмечать особые требования (аллергии, предпочтения в меню).

2. **Планировщик бюджета**  
   Система должна предоставлять инструмент для установки общего бюджета, распределения средств по категориям (площадка, кейтеринг, фотография и т.д.) и отслеживания фактических расходов с автоматическим расчётом остатка.

3. **Управление подрядчиками**  
   Система должна позволять добавлять информацию о подрядчиках (фотографы, флористы, музыканты), хранить их контактные данные, условия контрактов, даты оплаты и заметки.

4. **Система уведомлений**  
   Система должна автоматически отправлять email и push-уведомления о важных событиях: изменение статуса RSVP, приближающиеся сроки оплаты, напоминания о задачах.

5. **Планировщик рассадки**  
   Система должна предоставлять визуальный инструмент для создания схемы рассадки гостей за столами с учётом их предпочтений и взаимосвязей, с возможностью drag-and-drop перемещения гостей.

### Нефункциональные требования

1. **Производительность**  
   Время загрузки основной страницы не должно превышать 2 секунд при среднем интернет-соединении. API-запросы должны обрабатываться менее чем за 500 мс в 95% случаев.

2. **Масштабируемость**  
   Система должна поддерживать одновременную работу не менее 10 000 пользователей без деградации производительности. Архитектура должна позволять горизонтальное масштабирование.

3. **Безопасность**  
   Все данные пользователей должны быть защищены. Пароли хранятся в хешированном виде (bcrypt). Передача данных осуществляется по HTTPS. Реализована защита от SQL-инъекций, XSS и CSRF атак.

4. **Доступность**  
   Система должна обеспечивать доступность 99.9% (не более 8.76 часов простоя в год). Резервное копирование данных выполняется автоматически каждые 24 часа.

5. **Адаптивный дизайн**  
   Интерфейс должен корректно отображаться и функционировать на всех устройствах: настольные компьютеры (1920x1080+), планшеты (768x1024), смартфоны (375x667+).

## Этап 2. Архитектура

### Вариант 1: Монолитная архитектура

#### Схема системы

```
┌─────────────────────────────────────────────────────────┐
│                    Клиент (Browser)                     │
│  ┌──────────────────────────────────────────────────┐   │
│  │         React SPA (Frontend)                     │   │
│  │  - Pages (Guests, Budget, Vendors, Seating)      │   │
│  │  - State Management (Redux/Zustand)              │   │
│  │  - REST API Client                               │   │
│  └──────────────────────────────────────────────────┘   │
└────────────────────────┬────────────────────────────────┘
                         │ HTTPS / REST API
                         ▼
┌─────────────────────────────────────────────────────────┐
│              Сервер приложений (Node.js + Express)      │
│  ┌──────────────────────────────────────────────────┐   │
│  │  ┌─────────┐  ┌──────────┐  ┌──────────────┐     │   │
│  │  │ Auth    │  │ Guests   │  │ Budget       │     │   │
│  │  │ Module  │  │ Module   │  │ Module       │     │   │
│  │  └─────────┘  └──────────┘  └──────────────┘     │   │
│  │  ┌─────────┐  ┌──────────┐  ┌──────────────┐     │   │
│  │  │ Vendors │  │ Seating  │  │ Notifications│     │   │
│  │  │ Module  │  │ Module   │  │ Module       │     │   │
│  │  └─────────┘  └──────────┘  └──────────────┘     │   │
│  └──────────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────────┐   │
│  │  База данных (PostgreSQL)                        │   │
│  │  - Tables: users, guests, budgets, vendors, etc. │   │
│  └──────────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────────┐   │
│  │  Файловое хранилище (Local/S3)                   │   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

#### Плюсы

- ✅ **Простота разработки** — единая кодовая база, проще разрабатывать и отлаживать
- ✅ **Простота деплоя** — один артефакт для развёртывания, меньше инфраструктуры
- ✅ **Меньше операционных расходов** — не требуется оркестрация контейнеров
- ✅ **Проще транзакции** — единая БД упрощает обеспечение целостности данных
- ✅ **Быстрый старт** — минимальная настройка для начала разработки

#### Минусы

- ❌ **Ограниченная масштабируемость** — масштабирование только целиком, а не по модулям
- ❌ **Единая точка отказа** — сбой в одном модуле может повлиять на всю систему
- ❌ **Сложность сопровождения** — с ростом проекта код становится сложнее поддерживать
- ❌ **Привязка к стеку** — сложно использовать разные технологии для разных модулей
- ❌ **Длительная сборка** — с ростом проекта увеличивается время сборки и тестирования

### Вариант 2: Микросервисная архитектура

#### Схема системы

```
┌─────────────────────────────────────────────────────────┐
│                    Клиент (Browser)                     │
│  ┌──────────────────────────────────────────────────┐   │
│  │         React SPA (Frontend)                     │  │
│  └──────────────────────────────────────────────────┘   │
└────────────────────────┬────────────────────────────────┘
                         │ HTTPS
                         ▼
┌─────────────────────────────────────────────────────────┐
│                  API Gateway (Kong/Nginx)                │
│  - Маршрутизация запросов                                │
│  - Аутентификация и авторизация                          │
│  - Rate limiting                                         │
└────────┬────────┬────────┬────────┬─────────────────────┘
         │        │        │        │
         ▼        ▼        ▼        ▼
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│  Auth    │ │  Guest   │ │  Budget  │ │ Vendor   │
│ Service  │ │ Service  │ │ Service  │ │ Service  │
│ (Node.js)│ │ (Node.js)│ │ (Node.js)│ │ (Node.js)│
└────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘
     │            │            │            │
     ▼            ▼            ▼            ▼
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│  Auth    │ │  Guest   │ │  Budget  │ │  Vendor  │
│   DB     │ │   DB     │ │   DB     │ │   DB     │
└──────────┘ └──────────┘ └──────────┘ └──────────┘

┌─────────────────────────────────────────────────────────┐
│              Message Broker (RabbitMQ/Redis)             │
│  - Асинхронная коммуникация между сервисами              │
│  - Обработка уведомлений                                 │
└─────────────────────────────────────────────────────────┘
```

#### Плюсы

- ✅ **Независимое масштабирование** — каждый сервис масштабируется отдельно по нагрузке
- ✅ **Изоляция сбоев** — отказ одного сервиса не влияет на остальные
- ✅ **Технологическая гибкость** — можно использовать разные технологии для разных сервисов
- ✅ **Независимый деплой** — обновление одного сервиса без переразвёртывания всей системы
- ✅ **Командная автономность** — разные команды могут работать над разными сервисами

#### Минусы

- ❌ **Сложность разработки** — необходимо управлять взаимодействием сервисов
- ❌ **Сложность деплоя** — требуется оркестрация (Kubernetes, Docker Swarm)
- ❌ **Высокие операционные расходы** — больше инфраструктуры и мониторинга
- ❌ **Сложность отладки** — распределённая трассировка и логирование
- ❌ **Распределённые транзакции** — сложнее обеспечивать согласованность данных

## Сравнительный анализ и выбор

| Критерий | Монолит | Микросервисы |
|----------|---------|--------------|
| Сложность разработки | Низкая | Высокая |
| Масштабируемость | Ограниченная | Отличная |
| Стоимость инфраструктуры | Низкая | Высокая |
| Время выхода на рынок | Быстро | Медленнее |
| Надёжность | Средняя | Высокая |
| Поддержка команды | 1-3 разработчика | 3+ разработчиков |

### Рекомендуемый вариант: **Монолитная архитектура (Вариант 1)**

**Обоснование:**

Для портала планирования свадьбы на начальном этапе рекомендуется монолитная архитектура по следующим причинам:

1. **Скорость разработки** — проект можно реализовать и запустить значительно быстрее, что важно для проверки идеи на рынке
2. **Умеренные требования к масштабируемости** — на старте не ожидается десятков тысяч одновременных пользователей
3. **Снижение затрат** — меньше инфраструктуры и операционных расходов
4. **Проще поддерживать** — небольшой команде проще работать с единой кодовой базой

**Стратегия эволюции:** При росте нагрузки и пользовательской базы систему можно поэтапно декомпозировать в микросервисы, начиная с наиболее нагруженных модулей (например, уведомления или обработка гостей).
